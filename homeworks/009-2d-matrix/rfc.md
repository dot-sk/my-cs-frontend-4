## Универсальная двухмерная матрица на основе типизированного массива

Нужно сделать класс универсальной двумерной матрицы, поддерживающей view для кортежей формата RGBA (Red, Green, Blue, Alpha)

Нужно сравнить скорость сериализации и десериализации этой структуры с JSON-представлениями (плоский массив или массив массивов) при обработке изображения, получаемого из Canvas с помощью метода getImageData, а также нужно сравнить размер итогового файла на диске (с учётом сжатия)

### Результаты

Замеры на Apple M1 Max, Bun 1.3.13, mitata, [изображение](https://www.nasa.gov/image-detail/amf-art002e012090/) 1920×1280 RGBA. Бинарный формат: u32 width + u32 height + raw RGBA байты. JSON flat: массив из 9.83 млн чисел от 0 до 255. JSON nested: массив из 2.46 млн массивов по 4 числа

#### Размеры файлов

| формат         | raw      | gzip    | ratio |
| -------------- | -------- | ------- | ----- |
| Matrix2D (bin) | 9.38 МБ  | 2.33 МБ | 0.25  |
| JSON (flat)    | 35.89 МБ | 3.43 МБ | 0.10  |
| JSON (nested)  | 40.58 МБ | 3.74 МБ | 0.09  |

JSON в 4 раза тяжелее bin – из-за структурного мусора и текстового представления данных. gzip хорошо жмет повторы в тексте и плохо жмет картинки, поэтому jpg сжимает изображение не через словарь, а хранит только разницу между соседними пикселями, которая часто околонулевая. У bin формата байты распределены равномерно и алгоритму сжатия ловить почти нечего, но тем не менее gzip прилично сжал до 1/4 от оригинала. Думаю, что в первую очередь потому, что я выбрал изображение луны с большими участками монотонного цвета. На более сложных изображениях ratio будет хуже, на более простых – лучше. JSON же сжался примерно в 10 раз, потому что текстовое представление содержит много повторяющихся символов

Bin весит меньше всех (2.33 МБ против 3.43 МБ), потому что не содержит почти никакого оверхеда на хранение - только изображение и 8 байт заголовка

#### Время

| метрика     | bin     | flat (JSON) | nested (JSON) |
| ----------- | ------- | ----------- | ------------- |
| serialize   | 363 мкс | 82 мс       | 124 мс        |
| deserialize | 65 нс   | 157 мс      | 236 мс        |

Bin на serialize **~225× быстрее** JSON, на deserialize – **в миллионы раз**. Bin deserialize ничего не парсит по сути, только строит view поверх готового `ArrayBuffer`

Flat сериализуется быстрее nested (82 мс против 124 мс) при меньшем выходном тексте – меньше аллокаций промежуточных массивов и меньше структуры в JSON-выводе

#### Фан-факт: Array.from(uint8array) в 24× медленнее ручного цикла

Изначально я писал `data: Array.from(imageData.data)` и получил для flat **196 мс**, что было _медленнее_ nested. Это показалось странным – по идее flat должен был выиграть. Немного покопал, и обнаружил:

| подход                             | время      |
| ---------------------------------- | ---------- |
| `Array.from(PIXELS)`               | 130 мс     |
| `[...PIXELS]` (spread)             | 131 мс     |
| **ручной цикл `for (let i; ...)`** | **5.5 мс** |

Немного копнул cpu профайл и увидел следующее:

```
Array.from(typedArr) на каждый элемент:
  performIteration [из 405]
    └─ next [504]
        └─ arrayIteratorNextHelper [1789]
        ├─ typedArrayViewLength [2094] ← каждый раз
        └─ typedArrayViewIsDetached [794] ← каждый раз
        (потом возврат { value, done })

manual for-loop:
(inlined в caller) [527]
– все
```

То есть for loop JIT оптимизировал в SIMD инструкции, а `Array.from` и spread пошли через iterator – это 2698 сэмплов (1789 + 504 + 405) только на саму итерацию. На каждый из 9.8 млн пикселей – пять-семь функций в стеке. Спека [Array.from](https://tc39.es/ecma262/#sec-array.from) шаги 4–5 предписывают: если у source есть `[Symbol.iterator]`, идти через iterator protocol, а typed array такой Symbol имеет (через [%TypedArray%.prototype.values](https://tc39.es/ecma262/#sec-%25typedarray%25.prototype.values))

Попутно увидел, что на каждом шаге повторно проверяется длина буфера и detached-флаг. Это 2888 сэмплов (2094 + 794). Это требование спеки – на каждое чтение элемента typed array вызывается [TypedArrayGetElement](https://tc39.es/ecma262/#sec-typedarraygetelement), который первым шагом вызывает [IsValidIntegerIndex](https://tc39.es/ecma262/#sec-isvalidintegerindex), а тот первым шагом проверяет `IsDetachedBuffer(obj.[[ViewedArrayBuffer]])`. Дополнительно через [IsTypedArrayOutOfBounds](https://tc39.es/ecma262/#sec-istypedarrayoutofbounds) идет вторая проверка на detached, ведь пользователь действительно может вызвать `arrayBuffer.transfer()` посередине

Ручной цикл `for (let i = 0; i < n; i++) a[i] = src[i]` пишет напрямую по индексам и работает в **~24 раза быстрее**

После замены `Array.from` на ручной цикл flat обогнал nested как и ожидалось (82 мс vs 124 мс)

## Vector vs Array<object>

Cделал `Vector` – динамический массив с `push`/`pop`/`shift`/`unshift`/`view`. Сравнил с обычным `Array<{red, green, blue, alpha}>`, N = 100k операций на бенч, 1М на замер удержанной памяти

#### Скорость

| сценарий              | Vector  | Array<object> |
| --------------------- | ------- | ------------- |
| set N=100k            | 415 мкс | 466 мкс       |
| push N + pop N (100k) | 838 мкс | 778 мкс       |
| unshift N=10k         | 3.35 мс | 8.12 мс       |

`set` и `push+pop` идут практически вровень с нативным Array, что неожиданно – думал, что обертка через `entity.set` будет дороже. На `unshift` Vector в **2.4× быстрее**, потому что под капотом `Uint8Array.copyWithin` – это SIMD ускоренный метод копирования памяти, а `Array.prototype.unshift` сдвигает указатели на объекты в JS-цикле

#### GC pressure (аллокаций на итерацию бенча, из mitata)

| сценарий      | Vector | Array<object> |
| ------------- | ------ | ------------- |
| set 100k      | 31 кб  | 126 кб        |
| push+pop 100k | 24 кб  | 47 кб         |
| unshift 10k   | 33 кб  | 80 кб         |

Vector в **2-4× меньше мусора**

#### Удерживаемая память при 1М элементах

| контейнер     | heap     | arrayBuffers | итого    |
| ------------- | -------- | ------------ | -------- |
| Vector        | 0 МБ     | 4 МБ         | **4 МБ** |
| Array<object> | 60.66 МБ | 0 МБ         | 61 МБ    |

Vector в **15× компактнее**. Ожидаемо – у него один сплошной ArrayBuffer на 4 байта × 1M, у Array<object> миллион объектов с V8-оверхедом (hidden class pointer, prototype slot, padding под выравнивание). Но важнее не только размер – **major GC у Array идет по миллиону ссылок**, у Vector mark phase касается **одного** объекта

#### Когда Vector выигрывает в реальных задачах

| Сценарий                              | Выбор  | Почему                                               |
| ------------------------------------- | ------ | ---------------------------------------------------- |
| Кратковременный список внутри функции | Array  | per-push быстрее, GC соберет всю функцию целиком     |
| Долгоживущий буфер (10к+, минуты+)    | Vector | major GC будет в десятки раз быстрее                 |
| Передача в Canvas/WebGL/Audio API     | Vector | им нужен `Uint8ClampedArray`/`Float32Array` напрямую |
| Предсказуемый render loop             | Vector | предсказуемое время GC независимо от N               |
| Гибкая структура с разными элементами | Array  | у Vector жесткая схема элемента                      |

#### Фан-факт: view с замыканиями ел 40% CPU

Копнул перф через cpu профайлер. Изначально `RGBA.view` возвращал object literal с 4 геттерами и 4 сеттерами через замыкание над `bytes` и `offset`:

```ts
return {
  get red() {
    return bytes[offset];
  },
  set red(v) {
    bytes[offset] = v;
  },
  // ... ещё 6 акцессоров
};
```

В CPU-профиле было видно:

```
| Self% | Function       | Location        |
|-------|----------------|-----------------|
| 39.9% | view           | rgba.ts:42      | ← вот этот объектный литерал
|  7.8% | Vector.set     | 009-vector.ts   |
|  7.5% | RGBA.get       | rgba.ts:38      |
```

`view` сам по себе делал **94 нс на вызов**, остальной вектор – ~10 нс. Причина в том, что object literal с inline-геттерами создает **уникальный hidden class на каждый вызов** (V8 не может расшарить shape между разными closure environments) + 8 объектов окружения для замыканий = ~9 аллокаций на каждый `view(i)`

Заменил на класс с акцессорами на прототипе:

```ts
class RGBAColorRef implements RGBARef {
  constructor(
    public bytes: Uint8Array,
    public offset: number,
  ) {}
  get red() {
    return this.bytes[this.offset];
  }
  set red(v) {
    this.bytes[this.offset] = v;
  }
  // ... остальные через this
}
```

Стало **16 нс на вызов – в 5.8× быстрее**

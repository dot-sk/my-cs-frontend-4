## Разные форматы данных

### 1. **CSV vs JSON**

Реализуйте поточный (streaming) парсер CSV-файлов с заданным разделителем и сравнить его с нативной обработкой JSON

**Что нужно сделать:**

- Написать парсер, который читает CSV построчно (или по чанкам), не загружая весь файл в память.
- Протестировать на файле ≥10 МБ, сравнив с аналогичными данными в JSON (используя `JSON.parse`)
  - Измерить и сравнить:
    - Общее время обработки
    - Время до первой записи (latency до начала получения данных)
    - Пиковое потребление памяти
    - Размер данных в CSV и JSON БЕЗ и СО сжатием

- (со звёздочкой): Добавить поддержку экранирования разделителя (например, "a,b" или a\,b)

### 2. **CSV vs MessagePack**

Сравните ваш парсер CSV с MessagePack на больших файлах (файлы должны содержать одинаковый набор данных). Оба парсера должны работать в потоке, не загружая данные целиком в память. Проанализируйте результат также, как и для JSON.

Для работы с MessagePack используйте https://www.npmjs.com/package/msgpackr (с помощью PackrStream и UnpackrStream)

### Результаты

Замеры на Apple M1 Max, Bun 1.3.11, 20 итераций + 5 warmup, медиана, 1 500 000 записей `[id, timestamp, score, active, name, city]` (int + int64 + float + bool + 2 строки)

#### Размеры файлов

| формат      | raw       | gzip     | ratio |
| ----------- | --------- | -------- | ----- |
| CSV         | 109.12 МБ | 16.47 МБ | 0.15  |
| JSON        | 117.71 МБ | 16.63 МБ | 0.14  |
| MessagePack | 96.63 МБ  | 21.02 МБ | 0.22  |

#### Время

| метрика          | CSV     | JSON   | MessagePack |
| ---------------- | ------- | ------ | ----------- |
| до первой записи | 1.58 мс | 478 мс | 0.72 мс     |
| полный парсинг   | 589 мс  | 481 мс | 573 мс      |

JSON.parse умеет отдавать записи только все сразу, поэтому у него первая запись стоит столько же, сколько полный парсинг. Стриминговые реализации отдают первую запись за ~миллисекунду

На полном парсинге JSON выигрывает, потому что работает через нативную реализацию C++. msgpackr на JS медленнее на ~19%. Мой CSV парсер на ~22% медленнее JSON – сопоставимо с msgpackr

#### Память (пик rss)

| формат      | peak rss |
| ----------- | -------- |
| CSV         | 120 МБ   |
| JSON        | 797 МБ   |
| MessagePack | 108 МБ   |

JSON потребляет в пике **797 МБ** на файле в 118 МБ: ~120 МБ файла как строка + ~500 МБ распарсенный массив + рантайм

### Почему так

- **JSON не потоковый** – `JSON.parse` грузит и парсит все сразу. 800 МБ rss на 120 МБ файле, первая запись = полный парсинг
- **CSV и MessagePack потоковые** – держат только текущую запись, пик rss 108–120 МБ – это практически только рантайм, независимо от размера файла, первая запись за миллисекунду
- **Скорость парсинга определяется реализацией, а не форматом** – нативный JSON.parse быстрее JS-реализации msgpackr на ~19%, мой CSV на JS сопоставим с msgpackr
- **После gzip размеры сжимаются в 5–7 раз** у всех трех, но коэффициент сильно зависит от данных – у нас много повторов (одинаковые префиксы в `name`, 8 повторяющихся городов, монотонный timestamp), gzip на такое хорошо реагирует. На разнообразных данных (UUID-ы, случайный текст, бинарщина) сжатие было бы заметно слабее. Если хранение сжатое – выбор формата определяется не размером, а моделью потребления (стрим vs разовый parse)

### Как я ускорил парсер

Первая версия на 100 МБ данных работала за **2551 мс** и в 5 раз медленнее JSON.parse. Внутри она читала chunk посимвольно через `for (const char of chunk)`, собирала поле через `buffer += char` и делала 3 сравнения на каждый символ (QUOTE/SEP/LF). Количество итераций было основным боттлнеком

Я снял CPU profile через `bun --cpu-prof`, открыл в speedscope. Увидел, что больше всего времени самого парсера жрали две вещи:

- **`buffer += char`** – склеивание строки
- **Сравнения текущего символа с QUOTE/SEP/LF**

**Идея оптимизации:** логично было уменьшить количество операций. Например, вместо обхода каждого символа – найти позицию ближайшего спецсимвола через `String.indexOf`, скопировать все данные между cursor и этой позицией через `slice` в `buffer`, и таким образом сэкономить сотни или даже тысячи операций

**Ошибки по дороге:**

1. Сначала сделал парсер **медленнее**, выстрелил себе в ногу O(n^2) – сделал вызов `indexOf` на каждое поле и при отсутствии символа в данных сканировал весь остаток chunk впустую => добавил **кэш позиций** с обновлением только когда cursor долнал найденную позицию - сложность стала O(n/k), где k – среднее расстояние между спецсимволами. Это дало огромный прирост в производительности
2. Надо было часто вывысчитывать минимальный индекс, но indexOf() дает -1 когда пусто, поэтому поменял -1 на что-то большое. Пробовал `Infinity` как такой sentinel, но стало медленнее. Потому что `Infinity` это Float64, а остальные индексы маловероятно выходят за Int32 позиции, JIT оптимизирует их в **SMI**. Смешивание типов сломало JIT. Заменил на `chunk.length` (тоже скорее всего SMI) – все снова стало быстро

**Результат:** 2551 мс → **589 мс**. В ~4.3 раза быстрее, разрыв с JSON.parse закрылся с 5x до 1.2x. Пик rss тоже упал со 165 МБ до 120 МБ – меньше промежуточных строк от склеивания строк, меньше мусора для GC

**Важный урок:** без CPU-профиля я бы оптимизировал не то. Сначала думал, что основной прирост производительности даст работа с сырыми байтами. В целом и так можно было догадаться, но в профаклере боттлнек был очевиден

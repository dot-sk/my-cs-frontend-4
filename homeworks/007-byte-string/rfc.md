## Кодирования массива UTF-8 строк переменной длины

Надо реализовать класс/функцию для бинарной сериализации массива UTF-8 строк (можно использовать `TextEncoder` / `TextDecoder`).

**Формат:**

- 4 байта: количество строк (uint32)
- Для каждой строки:
  - 4 байта: длина строки в байтах (uint32)
  - N байт: UTF-8 данные строки

**Требования:**

- Поддержать произвольное количество строк
- Поддержать возможность чтения элемента массива по заданному индексу
- Обработать пустые строки

```js
const strings = ["hello", "мир", ""];

const buffer = encodeStrings(strings);

console.log(buffer.at(0)); // "hello"
console.log(buffer.at(-1)); // ""

const decoded = decodeStrings(buffer); // ["hello", "мир", ""]
```

## Кодирования массива UTF-8 строк переменной длины с помощью ссылок

Вариант, где длины и указатели на строки вынесены в отдельную секцию-индекс в начале буфера, а сами данные строк лежат в хвосте

**Формат:**

- 4 байта: количество строк (uint32)
- Для каждой строки (подряд в индексе):
  - 4 байта: длина строки в байтах (uint32)
  - 4 байта: указатель на начало данных строки в этом же буфере (uint32)
- После индекса – UTF-8 данные строк в произвольном порядке

**Требования:**

- Все то же, что в первом варианте
- Метод `set(idx, str)` для обновления строки по индексу

**Задание со звездочкой:**

Реализовать `set`, который обновляет строку внутри массива Простое решение – дописывать новую строку в хвост буфера и переписывать пару `(len, ptr)` в индексе, старые байты остаются мусором Это дает O(1) на set ценой фрагментации

```js
const strings = ["hello", "мир", ""];

const sb = StringBufferRef.fromData(strings);

console.log(sb.at(0)); // "hello"

sb.set(0, "Привет, ");

console.log(sb.at(0)); // "Привет, "

sb.set(-1, "!");

console.log(sb.toArray()); // ["Привет, ", "мир", "!"]
```

### Сравнение at() у обеих реализаций

- `StringBuffer` (первый вариант): схема `[count][len1][bytes1][len2][bytes2]...` В чистом виде на `at(i)` надо пройти по буферу с начала, читая длину каждой предыдущей строки и прыгая вперед – получается O(n) на каждый `at` Можно держать внешний `Uint32Array` с офсетами – тогда `at` становится O(1), но это уже доп. структура рядом с буфером
- `StringBufferRef` (второй вариант): схема `[count][(len,ptr)*count][data...]`, длины и указатели лежат прямо в буфере На `at(i)` считаем адрес индекса как `4 + i*8` и делаем два чтения `uint32` через `DataView` – O(1) без внешних структур

### Результаты

Замеры на Apple M1 Max, Bun 1.3.11, mitata 100_000 вызовов `at(i % size)` в каждой итерации

| размер массива | StringBuffer | StringBufferRef | разница  |
| -------------- | ------------ | --------------- | -------- |
| 100            | 80.46 ms     | 9.11 ms         | **8.8x** |
| 10_000         | 6.69 s       | 8.96 ms         | **747x** |

`StringBuffer` растет линейно с размером массива (от 80 мс на 100 строк до 6.7 секунд на 10к) – at() у него работает за O(n), итого O(n\*k) где `k` - количество вызовов. `StringBufferRef` держит около 9 мс вне зависимости от размера – at() у него работает за O(1)

### Почему так?

Формат с `(len, ptr)` в начале – это by-design O(1) рандомный доступ, без внешнего индекса

У формата с инлайн-длинами `[len1][bytes1][len2][bytes2]...` данные плотнее (нет 4 лишних байт указателя на каждую строку), но random access стоит прохода по всему буферу. Можно получить O(1), нужно платить памятью на отдельный индекс офсетов – по размеру это те же 4 байта на строку, просто лежат рядом, а не внутри
